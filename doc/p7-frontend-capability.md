# P7 前端工程师能力模型

> P7(阿里职级,对应腾讯 T9-T10、字节 2-2/3-1、Meta E5、Google L5)是「技术专家」级别。本文从能力模型出发,讲清楚 P7 到底要会什么、和 P6 的本质差别在哪、怎么往 P8 走。结合本项目(部署工程化)的具体场景举例。

## 1. P7 在职级体系中的位置

```
                技术深度
                  │
   P5  ──── 入门   │
   P6  ──── 熟练   │               影响力范围
   P7  ──── 专家 ─┼─────────────────  跨团队
   P8  ──── 资深   │               跨部门
   P9  ──── 总监   │               跨 BU / 行业
                  │
                业务驱动
```

P7 的核心定位:**能独立负责一个完整的技术领域,从设计到落地到长期演进,并对多个团队的工程效率产生正向影响**。

不是「写代码更快」,不是「懂更多框架」,而是**「定义问题 + 系统解决 + 放大影响」**。

## 2. P7 vs P6 的本质区别

| 维度 | P6(高级) | P7(专家) |
|---|---|---|
| **接到任务** | 给定方案,执行到位 | 给定问题,自己出方案 |
| **复杂度** | 单个模块/功能 | 跨模块/跨系统 |
| **时间视角** | 当前迭代 | 半年到一年演进 |
| **决策** | 技术选型 | 技术选型 + 权衡 + 长期代价 |
| **协作** | 组内 | 跨团队(后端/客户端/设计/产品) |
| **影响** | 自己写的代码 | 团队规范、工程基础设施 |
| **业务** | 理解需求 | 理解业务逻辑,能反向推动 |
| **遇到模糊** | 找 leader 确认 | 自己把模糊拆清楚 |

**最关键的差别**:P6 是「执行单元」,P7 是「问题 owner」。P6 被分派任务,P7 被分派问题。

举本项目的例子:
- **P6**:被告知「做一套灰度发布系统」,会按部就班做 KV 存储 + 规则匹配 + UI,功能跑通交付。
- **P7**:被告知「我们想让发布更安全、能灰度验证」,会先问「灰度验证的目标是什么?是降低上线故障率,还是支持 A/B 实验?」然后才决定做不做、做到什么程度、用这套技术栈还是别的。

## 3. 七大能力维度

### 3.1 技术深度(至少一个领域是专家级)

P7 不能哪个领域都「略懂」,必须有**至少一个细分领域**能说到行业前沿水平。可选方向:

- **框架/渲染**:React Fiber 调度、Vue 响应式原理、虚拟 DOM diff 优化、并发渲染
- **性能**:Core Web Vitals 优化、长任务分析、内存泄漏排查、首屏指标体系
- **工程化**:构建工具(Vite/Webpack/esbuild/turbo)、Monorepo、微前端、模块联邦
- **基础设施**:CDN/边缘计算/SSR/ISR、PWA、Service Worker、Web Workers
- **工具链**:AST(Babel/SWC)、Linter、TypeScript 编译器、Codemod
- **安全**:CSP、XSS/CSRF、SSO、SameSite、供应链安全
- **可访问性**:WCAG、ARIA、屏幕阅读器、键盘导航
- **可视化**:WebGL、Canvas、SVG、D3、图表引擎
- **音视频**:WebRTC、MSE、HLS、低延迟直播

**专家级**意味着:能读源码、能改源码、能在生产环境定位到框架层以下的问题、能写技术深度文章被同行认可、能在面试中考察别人这一方向。

**只到「会用」不算深度**。会用 React 不算 React 专家,得能解释 Fiber 架构为什么这么设计、能定位 Concurrent Mode 下的状态丢失 bug、能给 React 提 PR。

### 3.2 技术广度(能横向连接多个领域)

P7 必须能和后端、客户端、SRE、数据、安全团队**对话**,不一定要会写他们的代码,但要懂:

- **后端**:API 设计、数据库、消息队列、微服务、缓存策略
- **客户端**:WebView 通信、原生桥、跨端(RN/Flutter/小程序)
- **基础设施**:K8s、Docker、CI/CD、监控、日志、链路追踪
- **数据**:埋点、数据仓库、A/B 实验平台、指标体系
- **安全**:鉴权(OAuth/SAML/JWT)、加密、合规(GDPR/隐私)

**为什么必须广**?因为 P7 要做系统设计,系统设计本质是「在多个领域的交叉点找最优解」。本项目里:

- 灰度发布涉及前端 + Worker(边缘计算)+ R2(对象存储)+ KV(分布式存储)+ Sentry(可观测性)+ CI/CD(GitHub Actions)
- 想做对,必须每块都懂到「能判断选型合理性」的程度
- 不懂 KV 的人会以为「读-改-写」没并发问题,出生产事故
- 不懂 Sentry 的人会让 source map 泄露到 CDN

广度的标准:**能在白板上给后端/SRE 讲清楚自己方案在他们那一侧的影响**,而不是「我前端这边 OK,你们看着办」。

### 3.3 架构能力(系统设计)

P7 要能独立设计**跨多模块、跨多系统**的方案,核心能力:

- **问题拆解**:把模糊需求拆成可执行的技术子问题
- **边界划分**:哪些前端做、哪些后端做、哪些 BFF 做、哪些不做
- **技术选型**:在多个候选方案里做权衡(性能/成本/可维护性/团队熟悉度)
- **演进规划**:不是一次设计到位,而是分阶段、能回滚、能降级
- **风险识别**:并发、缓存、降级、容灾、安全、合规

**本项目示例**:灰度发布系统的架构决策。

| 决策点 | P6 思路 | P7 思路 |
|---|---|---|
| 灰度规则存哪 | Redis | KV(R2 同账号、免运维、最终一致够用) |
| 多版本怎么共存 | 覆盖更新 | `artifacts/{buildId}/` 隔离,旧版本不删 |
| HTML 缓存策略 | 长缓存 + 刷新 | 永不缓存(Worker 动态注入配置) |
| 灰度命中判断在哪 | 后端 | Worker(边缘、零延迟、不增加后端负担) |
| 灰度规则并发写 | 不考虑 | 识别为风险,标注「学习项目可接受,生产用 Durable Object」 |
| Sentry Release | 用 git sha | 用 BUILD_ID 串联构建/部署/监控五条链路 |

**架构能力的标准**:画一张图,能把所有组件、数据流、边界、降级路径讲清楚,并且每个决策都能说出「为什么不是另一种」。

### 3.4 工程能力(让团队高效)

P7 不只是自己效率高,要让**整个团队效率高**。具体体现:

- **构建体系**:CI/CD、构建优化(增量、缓存、并行)、产物分析
- **质量体系**:单元测试、E2E、覆盖率、类型检查、Lint、Code Review 流程
- **规范体系**:代码规范、提交规范(Conventional Commits)、Changesets、版本管理
- **监控体系**:错误监控、性能监控、告警阈值、SLO
- **工具链**:脚手架、CLI、内部 npm 包、文档站

**P6 关注自己写好代码,P7 关注团队写好代码**。

本项目里这些都已经做了:lint + type-check + test 三道闸、Sentry 错误捕获 + source map、产物体积分析、自动部署、灰度管理 UI。但还有大量可以加的:E2E 测试(Playwright)、性能基线、视觉回归测试、依赖更新自动化(Renovate)、PR 模板、技术文档站。

### 3.5 业务能力(技术服务业务)

P7 必须懂业务,否则技术做了一堆没用的事。具体能力:

- **业务理解**:知道业务模式、收入来源、核心指标、当前痛点
- **指标思维**:能用数据说话(DAU、转化率、留存、LTV、CAC)
- **ROI 思维**:技术投入和业务收益的关系,不为技术而技术
- **反向推动**:发现业务流程问题,用技术手段优化
- **优先级判断**:在多个需求里识别哪些是真重要、哪些是伪需求

**P6 是「需求来,我做完」**,P7 是「需求来,我先问这需求解决什么业务问题,值不值得做,有没有更省力的方案」。

本项目举例:
- P6 会问「灰度管理 UI 加什么功能?」
- P7 会问「我们为什么要做灰度?是因为上次发布出过事故吗?事故频率多少?灰度能把故障率降到多少?投入 2 周值不值?」

技术决策必须有业务逻辑支撑,否则就是自嗨。

### 3.6 影响力(跨团队 / 跨组织)

P7 的影响力必须**超出自己的小组**。体现:

- **技术布道**:在团队/公司内部分享、写技术文章、对外发声
- **规范制定**:推动团队/公司的技术规范、最佳实践
- **人才培养**:带 P5/P6,做 mentor,做 code review,做技术分享
- **开源贡献**:给开源项目提 PR、维护开源项目、参与技术社区
- **跨团队协作**:能拉通后端/客户端/数据/设计一起做事

**P6 影响自己写的代码,P7 影响一群人的工作方式**。

影响力的衡量标准:
- 团队里其他工程师遇到问题会来问你
- 你的方案/规范被别的团队采用
- 你带过的人升职了
- 外部能查到你的技术输出(博客/视频/开源)

### 3.7 软技能(沟通、决策、抗压)

P7 软技能比 P6 重得多,因为要做的事变复杂了:

- **沟通**:能把技术方案讲给非技术人(产品/业务)听懂,能把业务需求翻译成技术问题
- **向上管理**:能向 leader 清晰汇报进度、风险、决策点,不报喜不报忧
- **跨团队协作**:能和不同团队的利益方对齐目标,处理冲突
- **模糊性处理**:能在信息不全的情况下做决策,而不是「等想清楚再动」
- **技术决策**:在多个方案里做选择,并为决策负责
- **抗压**:线上事故冷静处理,deadline 压力下不糊弄

**最容易卡 P6 上不去 P7 的就是软技能**,尤其沟通和向上管理。技术很强的工程师如果只会埋头干,影响范围永远扩不大。

## 4. 具体场景下的能力体现

用本项目里可能的场景,看 P7 怎么处理。

### 场景 1:线上事故

**P6 反应**:看 Sentry 报错 → 找到出错的代码 → 修 → 上线。

**P7 反应**:
1. 第一时间止血(回滚到上个 buildId,1 分钟)
2. 拉群同步影响范围(哪些用户受影响、灰度命中没)
3. 定位根因(不是表面 bug,而是为什么会出这个 bug)
4. 修复 + 加测试覆盖这个 case
5. 复盘:为什么 CI 没拦住?为什么 code review 没发现?加什么机制能避免下次?
6. 输出事故报告,同步到团队,沉淀为 checklist

差别:**P7 不只解决这次的 bug,而是解决这一类 bug**。

### 场景 2:新需求评估

产品提:「给每个按钮加埋点,要统计点击率」。

**P6 反应**:列埋点字段,接埋点 SDK,逐个按钮加代码,提测。

**P7 反应**:
1. 问业务目标:统计点击率为了什么?优化转化?A/B 实验?老板要看?
2. 评估范围:所有按钮还是关键按钮?要不要全埋点(无代码侵入)?
3. 评估成本:逐个加 vs 全埋点方案,ROI 对比
4. 协调资源:埋点字段要和数据分析团队对齐,避免各团队自定义
5. 长期规划:埋点平台化,以后不用每个需求都改代码
6. 输出方案 + 排期 + 验证方法

差别:**P7 把「需求」当「问题」处理,不是当「指令」执行**。

### 场景 3:技术升级

要升级 React 18 到 React 19。

**P6 反应**:跑 `npm upgrade`,跑测试,修报错,上线。

**P7 反应**:
1. 评估必要性:React 19 有什么新特性是我们要用的?不升级会怎样?
2. 评估风险:breaking changes 列表、第三方依赖兼容性、并发渲染行为变化
3. 分阶段方案:先在 staging 跑、再灰度 1%、再全量
4. 监控预案:回滚机制、监控哪些指标(错误率、性能、关键页面)
5. 团队同步:新特性有哪些会影响日常开发,要不要更新规范
6. 时间窗口:避开业务高峰、避开大促

差别:**P7 把技术升级当项目管,不是当命令执行**。

## 5. 如何从 P6 成长到 P7

不是「年限够了就升」,是「能力到了才升」。建议路径:

### 5.1 选一个领域深钻

不要贪多,选**一个**细分领域,钻到能写文章、能给开源提 PR、能在面试中考察别人的程度。

- 选领域原则:团队需要 + 自己感兴趣 + 有应用场景
- 时间投入:每天 1-2 小时,持续 6-12 个月
- 验证标准:能做内部分享、能写系列文章、能解决生产环境难题

### 5.2 主动 owner 一件事

P6 的通病是「等人派活」。要破局,主动找一件**团队需要但没人管**的事,自己 owner 起来。

- 候选:技术债清理、监控体系建设、CI/CD 优化、新人 onboarding、性能优化、测试覆盖
- 关键:这件事要**有可见度**(团队能看到收益)+ **跨人协作**(不能只自己一个人干)
- 验证:做完了别人问你「这事是你 owner 吧」,而不是「这事谁做的来着」

### 5.3 跨团队项目

主动参与或发起跨团队项目,扩大协作面。

- 找后端聊 API 设计、找 SRE 聊部署、找数据聊埋点
- 不一定要发起大项目,小到「定一个 API 规范」也算
- 验证:别的团队的工程师知道你是谁、知道你负责什么

### 5.4 培养技术输出习惯

- 写文档:每个项目留一份设计文档(本项目已经做到了)
- 写博客:技术深度文章,不限于项目内
- 内部分享:定期在团队分享,练讲解能力
- 开源:给用过的库提 issue / PR,建立外部可见度

### 5.5 练模糊性处理

主动接**模糊的需求**,练「把模糊拆清楚」的能力。

- 产品说「页面感觉慢」,自己定义「慢」的标准、测量方法、目标值
- 老板说「我们做下监控」,自己设计监控什么、告警阈值、SLO
- 别人问「这方案行吗」,自己评估并给出明确 yes/no + 理由

这是 P7 最硬的能力,只能靠练。

### 5.6 找 mentor

找一个 P7+ 的工程师做 mentor,定期 review 你的方案、给反馈。

- 公司内:找资深同事
- 公司外:技术社区、前同事
- 关键:不是问「我该怎么做」,而是问「我这么做有什么没考虑到」

## 6. 反模式:什么样的工程师卡在 P6 上不去

| 反模式 | 表现 | 为什么卡住 |
|---|---|---|
| **只深度不广度** | 框架源码倒背如流,但不懂数据库 | 系统设计做不出来,影响范围扩不大 |
| **只广度不深度** | 啥都略懂,问深就露馅 | 没有不可替代性,什么都能被替代 |
| **埋头干不爱说** | 做了很多事,没人知道 | 影响力是 0,晋升答辩没材料 |
| **技术自嗨** | 用最新技术,业务没收益 | ROI 不成立,业务方不支持 |
| **等派活** | 不主动找事 | 没有 owner 表现,晋升没依据 |
| **怕模糊** | 需求不清晰就推回去 | P7 必须能把模糊拆清楚,做不到就还是 P6 |
| **不写文档** | 只有代码,没有设计文档 | 没法传承,影响力不扩散 |
| **不培养人** | 一个人干,不带人 | 影响力只限自己,放大不了 |
| **拒绝跨团队** | 「这是后端的事,不归我管」 | 系统边界扩不开,做不了架构 |
| **追新成瘾** | 每月换一个框架,没沉淀 | 没有深度积累,样样通样样松 |

## 7. P7 自检清单

下面 20 条,能稳做到 15 条以上,基本是 P7 水平。做到 10 条左右是 P6+。10 条以下还在 P5-P6 阶段。

- [ ] 至少一个前端细分领域能讲到源码级
- [ ] 能在白板上画出一个完整系统的架构(前端 + 后端 + 基础设施)
- [ ] 能给后端/SRE 讲清楚自己方案在他们那一侧的影响
- [ ] 接到模糊需求,能自己拆成技术子问题,不找 leader 确认
- [ ] 能做技术选型,并说出 3 个备选方案及权衡
- [ ] 做方案时会考虑降级、回滚、容灾
- [ ] 写的代码有测试覆盖,CI 能拦住常见错误
- [ ] 每个项目有设计文档,不只是代码
- [ ] 接过线上事故,写过复盘报告
- [ ] 主动 owner 过至少一个跨团队项目
- [ ] 带过 P5/P6,做过 mentor
- [ ] 在团队内做过技术分享,被认可
- [ ] 给开源项目提过 PR,或维护过开源项目
- [ ] 能把技术方案讲给产品/业务听懂
- [ ] 知道自己做的技术对应业务什么指标,能算 ROI
- [ ] 能在多个需求里识别优先级,敢说「这个不做」
- [ ] 遇到跨团队冲突能拉通对齐,不甩锅
- [ ] 能在信息不全时做决策,并为决策负责
- [ ] 团队里其他工程师遇到问题会来找你
- [ ] 离职 / 调岗后,你建立的体系还能继续运转

## 8. P7 到 P8 的下一步

P7 到 P8(资深专家 / Staff Engineer)的跃迁更难,核心差别:

| 维度 | P7 | P8 |
|---|---|---|
| 影响范围 | 跨团队 | 跨部门 / 跨 BU |
| 时间视角 | 半年到一年 | 一到三年 |
| 业务 | 理解业务 | 影响业务方向 |
| 技术 | 一个领域专家 | 多个领域专家,或一个领域行业顶尖 |
| 团队 | 带 1-3 人 | 带一个小团队 / 一个技术方向 |
| 决策 | 技术决策 | 业务 + 技术 + 组织的综合决策 |

P8 的核心能力是**「在更高维度做权衡」**:不只是技术方案选型,而是「这个方向值不值得做」「这个人该不该提」「这个业务要不要接」。

到了 P8,纯技术能力的边际收益下降,业务判断、组织能力、战略思考的权重上升。很多技术很强的 P7 卡在 P8 门口,通常是因为**只会用技术解决问题,不会用业务和组织解决问题**。

## 9. 总结

P7 前端工程师的能力模型可以浓缩成一句话:

> **在一个领域深到专家级,横向能连接多个领域,纵向能从业务到基础设施贯穿,能把模糊问题拆清楚并系统解决,能影响一群人的工作方式。**

技术深度是基础,广度是杠杆,架构是综合体现,工程能力是放大器,业务理解是方向舵,影响力是天花板,软技能是粘合剂。七个维度缺一不可,但权重因人因岗而异。

回到本项目:做这套部署工程化系统,对应到能力模型里——

- **技术深度**:Cloudflare 边缘计算、构建工具链、灰度发布引擎
- **技术广度**:Worker + R2 + KV + Sentry + GitHub Actions 横向打通
- **架构能力**:多版本共存、缓存分层、过渡期安全网、BUILD_ID 串联五条链路
- **工程能力**:lint/type-check/test/自动部署/监控 一体化
- **业务能力**:用灰度降低发布风险,服务业务稳定性
- **影响力**:文档化、可复用、可传承
- **软技能**:权衡决策(学习项目可接受 vs 生产环境方案)

把这个项目做到位,本身就涵盖了 P7 七大维度的大部分能力。剩下的差距主要在**业务理解**(项目偏工程,业务场景弱)和**影响力**(还是个人项目,没跨团队),这两块需要去真实工作环境里练。
